# トラブルシューティング

よくある問題と最初に試すべきこと。

## Reticulum が起動しない

**症状：** ログに `reticulum start: ...` の行が表示される、ディスカバリーが空、すべてのメッシュページが失敗する。

**確認事項：**

1. `about:` の設定パスがファイルの場所と一致しているか
2. `REN_BROWSER_CONFIG` または `--config` が有効なファイルを指しているか
3. インターフェース定義が構文的に正しいか
4. Ren Browser を実行しているユーザーが鍵とストレージパスを読み取れるか

アプリの外で Reticulum の設定を修正してから、設定から再読み込みするか再起動してください。

## ディスカバリーが空

[ディスカバリー](discovery.md) と [Reticulum の設定](reticulum-setup.md) を参照してください。

簡単なリスト：

- アナウンスが届くまで接続後しばらく待つ
- インターフェース上にピアが存在することを確認する
- 設定した UDP または TCP ポートのファイアウォールルールを確認する

## ページ読み込みタイムアウト

1. アドレスバーのハッシュを確認する
2. ディスカバリーから別のノードを開く
3. 設定のインターフェースで Reticulum がトラフィックを表示していることを確認する
4. メッシュネットワーク上の無線またはパスの変更後に再試行する

## データベースの破損または開けない

**症状：** プロファイルデータに関するエラー、データベースのリセットを提案される。

**オプション：**

1. バックアップから `renbrowser.db` を復元する（[データとプロファイル](data-and-profiles.md)）
2. UI からリセットする（ローカルのタブ、履歴、お気に入り、設定が削除される）
3. 壊れたファイルを名前変更して、Ren Browser に新しいデータベースを作成させる

Reticulum のアイデンティティはブラウザの DB リセットの影響を受けません。

## WASM または Micron パーサーエラー

Micron WASM の SRI チェックが失敗した場合：

1. チェックを無効化しない
2. 公式リリースから再インストールする
3. ソースからビルドした場合は、`frontend/dist/vendor/` を手動編集せずに `task build` を再実行する

## サーバーモード：空白ページまたは誤ったアセット

1. `--base-path` がリバースプロキシのマウントと一致しているか確認する
2. TLS がアップストリームで終端される場合は `--trust-proxy` を有効化する
3. Docker のポートマッピングを確認する（`-p 8080:8080`）

## サーバーモード：意図しない共有履歴

`--public-mode` で起動すると、各ブラウザが独自の `localStorage` コピーを保持します。

## 拡張機能が読み込まれない

1. マニフェストが `renbrowser.plugin.json` で有効な JSON である必要がある
2. `id` が `plugins/` 以下のフォルダー名と一致する必要がある
3. `engines.renbrowser` がアプリのバージョンを満たす必要がある
4. 不明な権限文字列は読み込み失敗を引き起こす

設定のエラー文字列を確認してください。

## Android ビルドが失敗する

1. `ANDROID_HOME` を設定する
2. `task android:install:deps` を実行する
3. [インストール](installation.md) に記載されているように API 34 と NDK r26 以降を使用する

## 開発：`task check` が失敗する

| エリア | コマンド |
|-------|---------|
| Go フォーマット | `task fmt:go` |
| Go テスト | `task test:go` |
| フロントエンド | `task frontend:check` |
| セキュリティスキャン | `task gosec` |

パッチを送る前に `task check` を実行してください。

## まだ解決しない場合

1. `about:` からバージョンをメモする
2. ターミナルまたは Docker からログをキャプチャする
3. メッシュコミュニティに質問するか、プロジェクトチャンネルを通じて詳細なバグレポートを送る

LXMF 経由のパッチ送信については [コントリビュート](contributing.md) を参照してください。
