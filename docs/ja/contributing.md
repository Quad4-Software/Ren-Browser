# コントリビュート

Ren Browser は Reticulum 経由で送られたパッチを受け付けます。GitHub のプルリクエストもプロジェクトのポリシーによっては歓迎されることがあります。このページは [CONTRIBUTING.md](../../CONTRIBUTING.md) に従っています。

## パッチのワークフロー

1. リポジトリをクローンまたはフォークする
2. ブランチを作成して集中した変更を行う
3. Go またはフロントエンドコードに触れた場合は `task check` を実行する
4. 明確なメッセージでコミットする
5. `git format-patch` でパッチをエクスポートする
6. `.patch` ファイルを LXMF で送信する

## LXMF でパッチを送る

デスティネーション：

```
f489752fbef161c64d65e385a4e9fc74
```

Sideband、Meshchat、MeshChatX、またはファイル添付をサポートする任意の LXMF クライアントを使用してパッチを添付します。メッセージ本文に短い説明を含めてください。

レビューはメッシュのペースで行われますので、しばらくお待ちください。

## エクスポートコマンド

```sh
# 最新のコミット
git format-patch -1

# 最後の N コミット
git format-patch -N

# main からのすべてのコミット
git format-patch main..HEAD
```

各コミットが 1 つの `.patch` ファイルになります。

## パッチのガイドライン

- 可能な限りパッチシリーズごとに 1 つの論理的な変更
- 送信前にテストする
- 既存のコードスタイルに合わせる
- 新しい Go ファイルに `// SPDX-License-Identifier: MIT` を保持する
- メッセージ本文で AI の使用を開示する（以下を参照）

## ライセンス

パッチを提出することで、[MIT ライセンス](../../LICENSE)の下でライセンスされることに同意します。作業を提出する権利があることを確認します。

## 生成 AI のポリシー

AI ツールを使用できる条件：

- セットアップがモデルに十分なコンテキストを提供している
- プロバイダーが貼り付けたコードをトレーニングに使用しない

[Reticulum Zen](https://reticulum.network/manual/zen.html) と [Reticulum ライセンス](https://reticulum.network/manual/license.html) を読んでください。

パッチメッセージで使用したツールを**開示**してください。意味のある方法で AI を使用しなかった場合は、簡潔にその旨を述べてください。

ローカルまたはオフラインモデルが強く推奨されます。

提出するすべてのものを読み、理解し、テストする必要があります。大量のレビューなしの出力は受け付けられません。

## セキュリティの問題

調整なしに LXMF 上でパッチとして脆弱性の詳細を送らないでください。[セキュリティ](security.md) のプロセスを使用してください。
