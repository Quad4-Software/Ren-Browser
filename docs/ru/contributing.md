# Участие в проекте

Патчи принимаются через Reticulum. См. также [CONTRIBUTING.md](../../CONTRIBUTING.md).

## Рабочий процесс

1. Клонируйте или форкните репозиторий
2. Ветка и сфокусированные изменения
3. `task check` при правках Go или фронтенда
4. Коммит с понятным сообщением
5. `git format-patch`
6. Отправка `.patch` по LXMF

## Отправка по LXMF

Назначение:

```
f489752fbef161c64d65e385a4e9fc74
```

Прикрепите патч через Sideband, Meshchat, MeshChatX или другой LXMF-клиент с вложениями. Кратко опишите изменение в теле сообщения.

Ревью идёт в ритме mesh. Наберитесь терпения.

## Команды экспорта

```sh
git format-patch -1
git format-patch -N
git format-patch main..HEAD
```

Один коммит : один файл `.patch`.

## Рекомендации

- Один логический change на серию патчей, где возможно
- Тестируйте перед отправкой
- Стиль кода как в репозитории
- `// SPDX-License-Identifier: MIT` в новых Go-файлах
- Раскрывайте использование ИИ (ниже)

## Лицензия

Отправляя патч, вы соглашаетесь на [MIT License](../../LICENSE) и подтверждаете право на публикацию кода.

## Политика генеративного ИИ

ИИ можно использовать, если:

- модель получает достаточно контекста
- провайдер не обучается на вашем коде

Прочитайте [Reticulum Zen](https://reticulum.network/manual/zen.html) и [лицензию Reticulum](https://reticulum.network/manual/license.html).

**Укажите** в сообщении к патчу, какие инструменты использовали. Если ИИ не применялся по сути, напишите это кратко.

Предпочтительны локальные офлайн-модели.

Вы обязаны прочитать, понять и протестировать всё отправленное. Массовый непросмотренный вывод не принимается.
